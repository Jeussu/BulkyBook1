# Somee Upload Steps

This checklist assumes the final local publish output is `publish-somee-final`. Do not upload source folders or local development files.

## 1. Local Pre-Upload

Run from the repository root:

```powershell
git status --short
dotnet build
dotnet build --no-incremental
powershell -NoProfile -ExecutionPolicy Bypass -File tools\qa\run-deployment-readiness.ps1 -SkipPublish -PublishDir "publish-somee-final"
```

Expected result: build succeeds, publish validation passes, and `appsettings.Development.json` is absent.

## 2. Upload These Files

Upload the contents of `publish-somee-final`, not the folder itself, to the Somee web root:

- `BulkyBookWeb.dll`, `BulkyBookWeb.exe`, `.deps.json`, `.runtimeconfig.json`.
- `web.config`.
- `appsettings.json` and `appsettings.Production.json`.
- all dependency `.dll` files and runtime folders generated by publish.
- `wwwroot` including `css`, `js`, `lib`, and `images/products/book-covers-modern`.

## 3. Do Not Upload

- repository source folders such as `BulkyBookWeb`, `BulkyBook.DataAccess`, `BulkyBook.Models`, `BulkyBook.Utility`.
- `.git`, `.vs`, `bin`, `obj`, local logs, screenshots, docs, or QA scripts.
- `appsettings.Development.json`.
- `migrate-somee.sql` unless you are applying it through Somee SQL tools, not the web root.
- any file containing real local credentials or personal publish settings.

## 4. Database Script Review

Generate the idempotent SQL script locally:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools\deployment\generate-somee-migration-script.ps1
```

Review `migrate-somee.sql` before applying it through Somee SQL tools or SSMS. Do not run `update-database` against Somee from this repo.

## 5. Environment Variables

Configure placeholders from `docs\deployment\SOMEE_ENVIRONMENT_VARIABLES.md` in Somee before first launch:

- `ASPNETCORE_ENVIRONMENT=Production`
- `ConnectionStrings__DefaultConnection=<somee-ms-sql-connection-string>`
- `Database__AutoMigrate=false`
- `SeedData__EnableDemoData=false`
- `Application__BaseUrl=https://<your-somee-domain>`
- `Stripe__EnableLocalCheckoutFallback=false`

Only add Stripe, Facebook, or SMTP secrets if those integrations are intentionally being tested.

## 6. Folder Permissions

Product uploads write under `wwwroot/images/products`. On Somee free hosting, verify the app can write to that folder before testing admin image upload. If write permission is unavailable, product upload may need a future external storage change.

## 7. First Launch Checks

- Open `https://<your-somee-domain>/Customer/Home/Index`.
- Open `/Identity/Account/Login` and `/Identity/Account/Register`.
- Confirm CSS and product cover PNGs load.
- Confirm no developer exception page appears.
- Confirm no local/demo/test copy is visible.
- Confirm `appsettings.Development.json` is not accessible.

## 8. Authenticated Checks

After DB, environment variables, and an admin account are configured:

- login as admin and open `/Admin/Product/Index`.
- open `/Admin/Product/Upsert?id=1` if product data exists.
- open `/Admin/Order/Index` if order data exists.
- login as a customer and test catalog/detail/cart only with staging-safe data.

Do not test real payments unless Stripe is configured with approved test keys and staging-safe data.

## 9. Static File Verification

Check these URLs after upload:

- `/css/site.css`
- `/js/product.js`
- `/js/order.js`
- `/js/company.js`
- `/images/products/book-covers-modern/aspnet-core-mvc-fundamentals.png`
- `/images/products/book-covers-modern/system-design-notes.png`

Each should return HTTP 200.

## 10. Rollback

- Keep a copy of the previous working publish output.
- If first launch fails, restore the previous web root contents.
- If SQL migration was applied and fails, use the Somee/SQL backup restore path; do not improvise schema changes through the app.
- Keep `Database__AutoMigrate=false` while investigating.
